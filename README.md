# Аналіз екологічної та радіаційної обстановки в зоні розташування АЕС України (2018–2025)

### Мета проекту

Дослідити динаміку радіоактивних викидів та скидів українських атомних електростанцій. Основна мета — виявити приховані закономірності в даних моніторингу, перевірити відповідність показників нормам безпеки та побудувати модель машинного навчання для прогнозування або класифікації екологічного стану в зонах розташування АЕС.

### Джерело даних

Дані отримані з Єдиного державного порталу відкритих даних:

* **Набір даних:** [Екологічна та радіаційна обстановка в зоні розташування атомних електростанцій](https://data.gov.ua/dataset/4a9d3d56-bd95-4c3e-97e7-1cdc7bcbd445)
* **Розпорядник:** Міністерство енергетики України

---

### Питання та гіпотези для аналізу

#### 1. Пошук аномальних відхилень у викидах

* **Питання:** Чи існують у часових рядах точки (квартали), коли рівень радіоактивних викидів був нетипово високим для конкретної АЕС?
* **Гіпотеза:** Показники можуть виходити за межі "нормального" шуму через технічні збої або особливості експлуатації, навіть якщо вони не перевищують гранично допустимі державні норми.

#### 2. Ідентифікація АЕС за «цифровим відбитком»

* **Питання:** Чи можна за хімічним складом викидів (співвідношенням $Cs^{137}$, $Co^{60}$ та йоду) точно визначити станцію без знання її назви?
* **Гіпотеза:** Кожна станція має унікальний профіль викидів, зумовлений типом реакторів (ВВЕР-440 vs ВВЕР-1000), віком обладнання та специфікою систем очистки.

#### 3. Факторний аналіз сумарного індексу викидів

* **Питання:** Які саме типи викидів (інертні гази, цезій чи йод) мають критичний вплив на формування загального індексу викидів (`index_radioactive_releas`)?
* **Гіпотеза:** Загальний екологічний ризик станції визначається не всіма показниками рівномірно, а домінуванням одного-двох специфічних факторів.

### Структура проєкту
Проєкт реалізовано як багатомодульну систему в Docker-середовищі:

```text
.
├── core/                       # Спільна логіка (менеджер БД)
├── data/                       # Вхідні дані
│   ├── raw/                    # Вихідний Excel-файл
│   └── test/                   # Дані для тестування
├── data_load/                  # Модуль ETL (завантаження в БД)
├── data_quality_analysis/      # Перевірка якості даних
├── data_research/              # ML-дослідження та статистика
├── db/                         # Сховище SQLite (Shared Volume)
├── notebooks/                  # Експерименти (EDA)
├── reports/                    # Результати (Shared Volume)
│   ├── figures/                # Графіки (PNG)
│   └── files/                  # Звіти (JSON)
├── src/                        # Вихідний код компонентів
├── tests/                      # Юніт-тести
├── visualization/              # Модуль генерації графіків
├── web/                        # Веб-інтерфейс (Streamlit)
├── docker-compose.yml          # Конфігурація розробки
└── docker-compose.prod.yml     # Production-конфігурація
```

---

### Опис сервісів та обмін даними
Система працює за принципом конвеєра (Pipeline), де результати передаються за допомогою **комбінованого підходу**:

1.  **data_load**: Зчитує XLSX, створює таблиці та імпортує дані в SQLite.
    * *Передача:* Записує файл у спільний том `./db`.
2.  **data_quality_analysis**: Аналізує повноту даних та дублікати.
    * *Отримання:* Читає БД з `./db`.
    * *Передача:* Записує JSON у спільний том `./reports/files`.
3.  **data_research**: Виконує статистичний аналіз та ML (пошук аномалій).
    * *Отримання:* Читає БД з `./db`.
    * *Передача:* Записує JSON у спільний том `./reports/files`.
4.  **visualization**: Будує графіки на основі ML-результатів.
    * *Отримання:* Читає JSON-звіти.
    * *Передача:* Записує PNG у спільний том `./reports/figures`.
5.  **web**: Відображає фінальні результати.
    * *Отримання:* Читає JSON та PNG з тому `./reports`.

### Команди запуску

#### 1. Підготовка
Переконайтеся, що файл даних лежить у папці:
`./data/raw/nuclear_safety_q4_2025.xlsx`

#### 2. Запуск усього проєкту
Для автоматичного складання образів та послідовного запуску всіх модулів виконайте:
```bash
docker compose up --build
```

#### 3. Перевірка результатів
Після завершення роботи розрахункових модулів (статус `Exited 0`), відкрийте браузер:
* **URL:** `http://localhost:8501`

#### 4. Зупинка та очищення
```bash
docker compose down -v
```

## Розгортання в хмарі (Azure + Terraform)

Цей розділ описує процес автоматизованого створення інфраструктури та запуску проєкту в Microsoft Azure.

### 1. Підготовка та відкриття Cloud Shell
1. Увійдіть у [Azure Portal](https://portal.azure.com).
2. Натисніть іконку **Cloud Shell** (`>_`) у верхній панелі керування.
3. Оберіть середовище **Bash**.
4. Згенеруйте SSH-ключі (якщо не робили цього раніше), щоб Terraform міг передати їх на віртуальну машину:
   ```bash
   ssh-keygen -t rsa -b 4096
   ```

### 2. Запуск інфраструктури (Terraform Apply)
1. Склонуйте репозиторій та перейдіть у папку з конфігурацією:
   ```bash
   git clone https://github.com/yanavyklyuk/open-data-ai-analytics.git
   cd open-data-ai-analytics/infra/terraform
   ```
2. Виконайте ініціалізацію, форматування, валідацію та розгортання:
   ```bash
   terraform init
   terraform fmt
   terraform validate
   terraform apply -auto-approve
   ```
3. Після завершення Terraform виведе `public_ip_address`. Скопіюйте цю адресу.

### 3. Перевірка результатів
Після виконання `apply` віртуальній машині потрібно **8–10 хвилин** для завершення внутрішнього налаштування (first boot configuration): оновлення системи, створення Swap-файлу та запуску Docker-контейнерів.

* **Веб-інтерфейс**: Відкрийте у браузері `http://<ВАШ_IP>:8501`.
* **Консольна перевірка**: 
    Підключіться до VM через SSH:
    ```bash
    ssh azureuser@<ВАШ_IP>
    ```
    Перевірте статус контейнерів:
    ```bash
    docker ps
    ```
    Ви маєте побачити 5 запущених сервісів: `nuclear_web`, `visualization`, `research`, `quality` та `data_load`.

### 4. Видалення ресурсів (Terraform Destroy)
Щоб уникнути зайвих витрат коштів на студентському рахунку Azure, після завершення роботи обов’язково видаліть інфраструктуру:
```bash
terraform destroy -auto-approve
```

## Моніторинг та спостережуваність 

Для відстеження стану інфраструктури, контейнерів та роботи ML-застосунку в проєкт інтегровано систему моніторингу на базі **Prometheus** та **Grafana**.

### 1. Як розгортається моніторинг
Моніторинг розгортається як окремий стек сервісів за допомогою Docker Compose. Він інтегрований у загальну інфраструктуру Azure через:
* **Terraform**: автоматично відкриває необхідні порти (3000, 9090) у мережевій групі безпеки (NSG).
* **Docker Network**: усі компоненти підключені до існуючої мережі проєкту `app_nuclear_network`, що дозволяє Prometheus збирати метрики з контейнерів за їхніми іменами.
* **Запуск**: виконується командою:
  ```bash
  docker-compose -f monitoring/docker-compose.monitoring.yml up -d
  ```

### 2. Сервіси збору метрик
Система збирає дані з чотирьох основних джерел:
* **Node Exporter**: збирає метрики хостової ОС (Azure Linux VM) — завантаження процесора, пам'яті, дискову та мережеву активність.
* **cAdvisor**: аналізує роботу Docker-контейнерів, показуючи споживання ресурсів кожним сервісом проєкту (`nuclear_web`, `db` тощо).
* **Prometheus**: виступає центральним агрегатором, який опитує всі цілі та зберігає дані в Time Series Database.
* **AES App (Python)**: веб-застосунок Streamlit експортує власні метрики (перегляди сторінок, кількість виявлених аномалій) на порту `8000` за допомогою бібліотеки `prometheus-client`.

### 3. Мережеві порти
Для роботи системи в Azure відкриті такі порти:
* **3000**: Веб-інтерфейс **Grafana** (візуалізація).
* **9090**: Веб-інтерфейс **Prometheus** (перевірка статусів збору даних — Targets).
* **8000**: Внутрішній порт метрик застосунку (доступний для Prometheus).

### 4. Доступ до Grafana
Для перегляду аналітики:
1. Відкрийте браузер за адресою: `http://<YOUR_VM_IP>:3000`.
2. Введіть стандартні облікові дані (якщо не було змінено):
   * **Логін**: `admin`
   * **Пароль**: `admin`
3. Джерело даних (Prometheus) підключено за адресою `http://prometheus:9090`.

### 5. Опис створеного дашборду
У Grafana налаштовано комплексний дашборд **"AES Monitoring System"**, який містить такі панелі:
* **CPU VM**: поточне завантаження процесора віртуальної машини у відсотках.
* **RAM Usage**: обсяг використаної оперативної пам'яті (GiB).
* **Running Containers**: кількість активних Docker-контейнерів у системі.
* **Disk & Network**: динамічні графіки швидкості читання з диска та мережевого трафіку.
* **App Performance**: використання CPU конкретно контейнером `nuclear_web`.
* **Business Metrics**: загальна кількість переглядів сторінок моніторингу АЕС користувачами.
